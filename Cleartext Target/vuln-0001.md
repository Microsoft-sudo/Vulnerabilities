# Unauthorized Error Event Injection via Exposed Sentry DSN

**ID:** vuln-0001
**Severity:** HIGH
**Found:** 2026-03-18 01:47:08 UTC
**Target:** sentry-v2.clear.in
**Endpoint:** /api/10/store/
**Method:** POST
**CWE:** CWE-798
**CVSS:** 8.6

## Description

A Sentry DSN (Data Source Name) was discovered exposed in the JavaScript bundle of the Clear application at https://app.clear.in/js/main.f94c7beddc8ba605a150.js. The DSN (https://607fd3b42fc9b74117f75a6900f89b00@sentry-v2.clear.in/10) allows unauthorized attackers to submit arbitrary error events to the organization's production error tracking system.

Sentry is an error tracking platform used by development teams to monitor application errors and exceptions. The exposed DSN includes a public key that grants write access to project 10 of the Sentry instance hosted at sentry-v2.clear.in.

The vulnerability enables attackers to:
1. Flood the error tracking system with fabricated events
2. Insert misleading error messages to confuse development/operations teams
3. Potentially overwhelm the Sentry instance if no rate limiting is in place
4. Submit events containing fabricated security alerts or false-positive detections

## Impact

Successful exploitation allows an attacker to:

- **Disrupt Monitoring and Alerting**: Flooding Sentry with false events can obscure genuine errors, delaying incident response and troubleshooting.
- **Create Operational Confusion**: Malicious actors can inject fabricated security-related errors or misleading diagnostic information, causing unnecessary investigations.
- **Potential Denial of Service**: If the Sentry instance lacks rate limiting (as observed), sustained event injection could degrade performance or exhaust resources.
- **Information Pollution**: The integrity of the error tracking data becomes compromised, reducing its reliability for debugging and monitoring.

Business impact includes reduced operational efficiency, delayed incident response, potential resource exhaustion, and decreased confidence in monitoring systems.

## Technical Analysis

The vulnerability stems from a hardcoded Sentry DSN exposed in client-side JavaScript. Sentry DSNs are intended to be kept secret as they provide write access to error tracking projects. The discovered DSN format is: `https://{PUBLIC_KEY}@{HOST}/{PROJECT_ID}`.

Technical findings:
1. **Successful Event Injection**: Unauthorized POST requests to `/api/10/store/` with the correct Sentry authentication header (`X-Sentry-Auth: Sentry sentry_key=607fd3b42fc9b74117f75a6900f89b00, sentry_version=7`) are accepted (HTTP 200).
2. **Project Enumeration**: Testing project IDs 1-14 revealed project 10 is accessible, while others return 403 Forbidden, indicating the exposed key has specific project permissions.
3. **No Rate Limiting Observed**: Multiple rapid submissions (5+ events in quick succession) were accepted without rate limiting responses (no 429 status codes).
4. **Information Disclosure**: The `/api/0/` endpoint returns `{"version":"0","auth":null,"user":null}`, revealing Sentry API structure.
5. **Event Confirmation**: Each successful submission returns a unique event ID (e.g., `{"id":"8e98cdbc09994559abdb40320240dbc5"}`), confirming event storage.

## Remediation

1. **Immediate Actions**:
   - Rotate the exposed Sentry DSN key to invalidate the compromised credential.
   - Implement IP whitelisting on the Sentry instance to restrict access to trusted sources only.
   - Review Sentry access logs for any unauthorized event submissions.

2. **Short-Term Measures**:
   - Remove hardcoded DSN values from client-side JavaScript bundles.
   - Implement strict rate limiting on the Sentry event ingestion endpoints.
   - Configure Sentry to reject events from unrecognized sources or with suspicious patterns.

3. **Long-Term Improvements**:
   - Establish a secret management solution for storing and distributing sensitive credentials.
   - Implement DSN rotation as part of regular security maintenance.
   - Add monitoring and alerting for anomalous event submission patterns (sudden spikes, unusual sources).
   - Conduct regular security reviews of client-side code for exposed credentials.

4. **Developer Education**:
   - Train developers on secure credential handling and the risks of hardcoding sensitive data.
   - Establish clear guidelines for where DSNs and other API keys should be stored (server-side only).

5. **Monitoring and Validation**:
   - Set up alerts for failed authentication attempts to Sentry endpoints.
   - Regularly audit which systems and services have access to error tracking.
   - Consider implementing a proxy or gateway for all Sentry traffic to enforce security controls.
