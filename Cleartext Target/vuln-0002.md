# Systemic Sentry DSN Exposure Across Multiple Application Modules

**ID:** vuln-0002
**Severity:** CRITICAL
**Found:** 2026-03-18 01:51:16 UTC
**Target:** sentry-v2.clear.in
**Endpoint:** Multiple Sentry API endpoints
**Method:** POST
**CWE:** CWE-798
**CVSS:** 10.0

## Description

Multiple Sentry DSNs (Data Source Names) have been discovered exposed across various JavaScript modules of the Clear application. This represents a systemic security issue where hardcoded Sentry credentials are exposed in client-side code, allowing unauthorized access to multiple error tracking projects.

The vulnerability affects numerous micro-frontend modules, each with their own dedicated Sentry DSN and project. Attackers can inject arbitrary error events into all of these projects, potentially flooding the entire error monitoring infrastructure.

Exposed DSNs discovered:
- Project 10: Main application (https://607fd3b42fc9b74117f75a6900f89b00@sentry-v2.clear.in/10)
- Project 38: Accounts Payable module (https://787bea27f1e848d4992ef6694c87f899@sentry-v2.clear.in/38)
- Project 75: Clear Capture module (https://45215690a2600c70cab25680166932aa@sentry-v2.clear.in/75)
- Project 40: Clear Compass module (https://2f0e69b8287e26b8c421df2f65a41d87@sentry-v2.clear.in/40)
- Project 30: Clear Data Pulls module (https://5487d8cc0b22ea4b461d19326d9b61e8@sentry-v2.clear.in/30)
- Project 39: Compliance Governance module (https://c33041c26c79563454b21db48c44ba60@sentry-v2.clear.in/39)
- Project 41: Contact Master module (https://7afea26d84ca0a8e83512a470e32664f@sentry-v2.clear.in/41)

Each DSN provides write access to a separate Sentry project, multiplying the attack surface and potential impact.

## Impact

The systemic exposure of multiple Sentry DSNs significantly increases the security risk:

- **Comprehensive Monitoring Disruption**: Attackers can flood all error tracking projects simultaneously, completely disrupting the organization's monitoring and alerting capabilities.
- **Cross-Module Pollution**: Malicious events can be injected into specialized modules (compliance, payments, data processing), creating false alerts across all business domains.
- **Amplified Denial of Service**: With no rate limiting observed, attackers can overwhelm the entire Sentry infrastructure by targeting multiple projects concurrently.
- **Operational Chaos**: Development and operations teams across all product areas would receive misleading error reports, hindering genuine incident response.
- **Business-Wide Impact**: The vulnerability affects core financial systems (payments, compliance, data processing), potentially impacting regulatory reporting and financial operations.

The systemic nature means a single attacker could compromise error tracking for the entire organization's application ecosystem.

## Technical Analysis

The vulnerability represents a systemic architectural issue where sensitive credentials are hardcoded in client-side JavaScript across multiple micro-frontend modules. Each application module maintains its own Sentry DSN, exposing the entire error tracking infrastructure.

Technical findings:
1. **Multiple Vulnerable DSNs**: 7 distinct Sentry DSNs discovered across different JavaScript modules, each with unique public keys and project IDs.
2. **All DSNs Vulnerable**: Every tested DSN accepted unauthorized event submissions (HTTP 200 responses with event IDs).
3. **No Rate Limiting**: All projects accepted rapid succession of events without rate limiting responses.
4. **Architectural Pattern**: The micro-frontend architecture appears to use a pattern where each module bundles its own Sentry configuration, multiplying the exposure.
5. **Source Locations**: DSNs found in various remote.js files: main.f94c7beddc8ba605a150.js, enterprise-shell-remote.js, accounts-payable-remote.js, clear-capture-remote.js, clear-compass-remote.js, clear-data-pulls-remote.js, compliance-governance-remote.js, contact-master-remote.js, e-invoice-enrich-remote.js.

The pattern suggests this is an organizational practice rather than an isolated developer error, indicating a need for architectural-level remediation.

## Remediation

1. **Immediate Emergency Actions**:
   - Rotate ALL exposed Sentry DSN keys simultaneously to invalidate compromised credentials.
   - Implement immediate IP whitelisting on the Sentry instance to restrict access to trusted production environments only.
   - Review Sentry access logs for any unauthorized event submissions across all projects.

2. **Architectural Remediation**:
   - Remove ALL hardcoded DSN values from client-side JavaScript bundles.
   - Implement a centralized configuration service for sensitive credentials.
   - Establish DSN management as a server-side concern only.
   - Conduct comprehensive security review of all micro-frontend modules.

3. **Security Controls**:
   - Implement strict rate limiting on all Sentry API endpoints.
   - Configure Sentry to reject events from non-whitelisted sources.
   - Add monitoring for anomalous event patterns across all projects.
   - Establish DSN rotation as part of regular security maintenance.

4. **Development Practices**:
   - Institute mandatory security reviews for all client-side code.
   - Implement automated scanning for exposed credentials in build pipelines.
   - Establish clear guidelines prohibiting hardcoded credentials in frontend code.
   - Train developers on secure credential management practices.

5. **Monitoring and Validation**:
   - Set up alerts for failed authentication attempts across all Sentry endpoints.
   - Regularly audit which systems have access to error tracking.
   - Implement a credential management solution with automatic rotation.
   - Conduct periodic security assessments of the micro-frontend architecture.
