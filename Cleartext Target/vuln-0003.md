# Session Fixation Vulnerability in ClearTax Authentication System Allows Account Takeover

**ID:** vuln-0003
**Severity:** HIGH
**Found:** 2026-03-18 01:58:40 UTC
**Target:** https://app.clear.in
**Endpoint:** /accounts/v2
**Method:** GET, POST
**CWE:** CWE-384
**CVSS:** 8.1

## Description

A critical session fixation vulnerability was identified in the authentication system of app.clear.in. The application accepts arbitrary session identifiers (ssoAId cookies) without proper validation, allowing attackers to set session IDs for victims before they authenticate.

When a victim subsequently authenticates using the attacker's pre-set session ID, the attacker gains full access to the victim's account. This vulnerability affects the main production environment (app.clear.in) but was found to be properly mitigated in the sandbox environment (app-sandbox.clear.in).

The vulnerability was confirmed through multiple test scenarios demonstrating that arbitrary, predictable, and concurrent session identifiers are accepted by the authentication system.

## Impact

Successful exploitation allows complete account takeover, providing unauthorized access to sensitive financial and tax information. In a business context like ClearTax which handles financial data, this could lead to exposure of confidential tax returns, business financial data, GST information, and other sensitive documents. The impact is particularly severe given the financial nature of the application.

Attack Scenario:
1. Attacker sets a session cookie (ssoAId) for the victim
2. Victim visits the application and authenticates normally
3. Attacker uses the same session cookie to access victim's account
4. Attacker gains full access to victim's financial and tax data

## Technical Analysis

The vulnerability stems from improper session management in the Single Sign-On (SSO) authentication system. The application uses a session identifier cookie named 'ssoAId' which is not properly validated during the authentication flow. 

Key findings:
- Arbitrary session IDs are accepted and persisted
- No server-side validation prevents session fixation
- Session IDs can be predictable (tested patterns like 'admin', 'test', '00000000-0000-0000-0000-000000000000' were accepted)
- Multiple concurrent sessions are possible
- The vulnerability exists in production but is correctly mitigated in the sandbox environment

The server responds with Set-Cookie headers confirming acceptance of custom session IDs, and these sessions remain active and valid for an extended period (1 year based on cookie expiration).

## Proof of Concept

To reproduce the session fixation vulnerability:

1. Open a browser or make an HTTP request to https://app.clear.in/accounts/v2 with a custom ssoAId cookie:
   ```
   GET /accounts/v2 HTTP/1.1
   Host: app.clear.in
   Cookie: ssoAId=attacker-controlled-session-id
   ```

2. The server will accept the custom session ID and respond with:
   - Status code 200
   - Set-Cookie header confirming the session ID
   - Full access to the authentication flow

3. If a victim authenticates using this pre-set session ID, the attacker can reuse the same session ID to access the victim's account.

Alternative reproduction using Python:
```python
import requests

# Set arbitrary session ID
session_id = "attacker-session-12345"
cookies = {"ssoAId": session_id}

    "https://app.clear.in/accounts/v2",
    cookies=cookies

# Session ID is accepted (status 200)
print(f"Status: {response.status_code}")
```

## Remediation

1. Implement proper session management:
   - Generate random, unpredictable session identifiers server-side
   - Regenerate session ID after authentication (session rotation)
   - Validate session creation timestamps and enforce expiration

2. Add session fixation protection:
   - Reject externally provided session IDs
   - Implement session binding to IP address or user agent
   - Use secure, HTTP-only cookies with proper flags

3. Enhance authentication flow:
   - Create new session after successful authentication
   - Invalidate pre-authentication sessions
   - Implement proper logout functionality that clears all session data

4. Security headers and cookies:
   - Ensure Secure and HttpOnly flags are set on session cookies
   - Implement SameSite cookie attribute
   - Set appropriate expiration times for session cookies

5. Monitoring and logging:
   - Log session creation and authentication events
   - Monitor for suspicious session patterns
   - Implement rate limiting on session creation
