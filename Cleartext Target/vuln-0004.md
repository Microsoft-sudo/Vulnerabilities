# Information Disclosure Exposing Microservices Architecture and Version Details

**ID:** vuln-0004
**Severity:** HIGH
**Found:** 2026-03-18 02:08:29 UTC
**Target:** https://app.clear.in
**Endpoint:** /ingestion/latest/product_release_metadata.json
**Method:** GET
**CWE:** CWE-200
**CVSS:** 7.5

## Description

A critical information disclosure vulnerability was identified in the ClearTax application that exposes detailed version information of multiple microservices and frontend components. The vulnerability exists in both production (app.clear.in) and sandbox (app-sandbox.clear.in) environments.

The application exposes a JSON endpoint at `/ingestion/latest/product_release_metadata.json` that returns version information for all deployed services. This endpoint is publicly accessible without authentication and reveals the complete microservices architecture, including specific version numbers for each component.

This information disclosure violates the principle of information hiding and provides attackers with valuable reconnaissance data that can be used for targeted attacks against known vulnerabilities in specific software versions.

## Impact

Successful exploitation allows attackers to:
- Map the complete application architecture and identify all deployed services
- Determine exact software versions for target development
- Research version-specific vulnerabilities and exploits
- Plan targeted attacks against known vulnerabilities in specific versions
- Understand the technology stack and identify weak components

The exposed information significantly reduces the reconnaissance effort required for attackers and increases the risk of successful exploitation by enabling more precise targeting of known vulnerabilities.

## Technical Analysis

The vulnerability exists due to improper access controls on the `/ingestion/latest/product_release_metadata.json` endpoint. The endpoint is served from the application's static hosting (AWS S3 behind CloudFront) and is publicly accessible without authentication.

Analysis of the exposed data reveals:
1. Production uses version 1.513.5 across all services
2. Sandbox uses version 1.448.0 across all services 
3. At least 14 distinct microservices/frontend modules are deployed
4. The architecture follows a microservices pattern with separate modules for different business functions

The endpoint responds with JSON containing key-value pairs where keys are module names and values are version numbers. This information is particularly valuable for attackers as it provides a complete technology inventory of the application ecosystem.

## Remediation

1. **Immediate Action**: Remove or restrict access to the `/ingestion/latest/product_release_metadata.json` endpoint
   - Implement authentication requirements for accessing version information
   - Move version metadata to internal-only endpoints not exposed to the internet
   - Serve the endpoint with HTTP 403 Forbidden or 404 Not Found responses

2. **Access Control Implementation**:
   - Add authentication and authorization checks for all metadata endpoints
   - Implement IP whitelisting for internal monitoring tools that need this data
   - Use network-level restrictions to prevent external access

3. **Information Minimization**:
   - Only expose minimal necessary information in any public endpoints
   - Avoid exposing specific version numbers in production environments
   - Use generic identifiers instead of exact version strings

4. **Monitoring and Detection**:
   - Implement logging and alerting for access to sensitive information endpoints
   - Monitor for reconnaissance attempts targeting version information
   - Regularly audit all endpoints for information disclosure risks

5. **Security Review**:
   - Conduct a comprehensive security review of all static and metadata endpoints
   - Implement automated scanning for information disclosure vulnerabilities
   - Review the Content Security Policy and other security headers
