# Session Fixation Vulnerability in ClearTax /save/ Module API Cookie Handling

**ID:** vuln-0009
**Severity:** MEDIUM
**Found:** 2026-03-18 14:16:03 UTC
**Target:** https://cleartax.in/save/api/
**Endpoint:** /save/api/
**Method:** ANY (GET, POST, HEAD, etc.)
**CWE:** CWE-384
**CVSS:** 5.4

## Description

A session fixation vulnerability was discovered in the ClearTax /save/ module API endpoint (/save/api/). The application's session management system accepts arbitrary 'aI' cookie values without validation or sanitization, allowing attackers to pre-set session identifiers.

The 'aI' cookie, which appears to be used for anonymous session tracking, accepts any client-provided value including malicious payloads such as path traversal attempts (../../etc/passwd), SQL injection attempts (injected'), and XSS payloads (<script>alert(1)</script>). The server does not validate, sanitize, or regenerate cookie values, enabling session fixation attacks where an attacker can force a user to use a predetermined session identifier.

This vulnerability is particularly concerning because the cookie is marked as HttpOnly, suggesting it's used for session tracking purposes. While the exact impact depends on how this cookie is used elsewhere in the application, session fixation typically enables session hijacking when combined with other attack vectors.

## Impact

Successful exploitation of this vulnerability could allow:

1. **Session Fixation**: Attackers can pre-set session identifiers and trick users into using them, potentially leading to session hijacking.

2. **Session Manipulation**: If the 'aI' cookie is used for authentication or authorization decisions elsewhere in the application, an attacker could manipulate session state.

3. **Privilege Escalation**: If combined with other vulnerabilities, this could enable unauthorized access to user accounts or administrative functions.

4. **Account Takeover**: In worst-case scenarios, successful session fixation could lead to complete account compromise.

The impact is limited by the requirement for user interaction (victim must use the attacker's cookie) and depends on how the 'aI' cookie is utilized throughout the application ecosystem. However, the lack of cookie validation represents a fundamental flaw in session management security.

## Technical Analysis

The vulnerability exists in the Express.js-based backend serving the /save/ module API endpoints. Analysis reveals:

1. **Cookie Handling**: The 'aI' cookie (likely "anonymous ID") is set with HttpOnly flag but accepts any client-provided value without validation.

2. **No Validation**: The server does not validate cookie format, length, or content. Malformed values including injection payloads are accepted.

3. **No Sanitization**: No input sanitization is performed on cookie values before storage or processing.

4. **No Regeneration**: Session identifiers are not regenerated upon authentication events, maintaining fixation vulnerability.

5. **Information Disclosure**: The endpoint also leaks rate limiting information (x-ratelimit-limit: 101, x-ratelimit-remaining: 100) which could aid attackers in planning attacks.

Testing demonstrates the server accepts:
- Normal values: test-session-fixation-123
- Path traversal: ../../etc/passwd  
- SQL injection: injected'
- XSS payloads: <script>alert(1)</script>

All values are accepted without error or sanitization, indicating complete lack of input validation for the 'aI' cookie parameter.

## Proof of Concept

To reproduce the session fixation vulnerability:

1. Send a request to the /save/api/ endpoint with a malicious 'aI' cookie value:
```bash
curl -I "https://cleartax.in/save/api/" -H "Cookie: aI=attacker-controlled-session-id"
```

2. Observe the server accepts any cookie value without validation:
```bash
# Test with various malicious payloads
curl -I "https://cleartax.in/save/api/" -H "Cookie: aI=<script>alert(1)</script>"
curl -I "https://cleartax.in/save/api/" -H "Cookie: aI=../../etc/passwd"
curl -I "https://cleartax.in/save/api/" -H "Cookie: aI=injected'"
```

3. All requests return HTTP 404 with the same rate limiting headers, confirming cookie acceptance:
```
HTTP/1.1 404 Not Found
Content-Type: application/json
x-ratelimit-limit: 101
x-ratelimit-remaining: 100
set-cookie: aI=<injected-value>;Path=/;Expires=...;Max-Age=63072000;HttpOnly
```

The server echoes back whatever cookie value is provided in the Set-Cookie header, demonstrating the session fixation vulnerability.

## Remediation

1. **Implement Cookie Validation**:
   - Validate 'aI' cookie format (UUID v4 expected based on observed patterns)
   - Reject malformed or suspicious cookie values
   - Implement length and character set validation

2. **Session Regeneration**:
   - Regenerate session identifiers upon authentication events
   - Implement session expiration and invalidation policies
   - Use cryptographically secure random session IDs

3. **Input Sanitization**:
   - Sanitize all cookie values before processing or storage
   - Implement proper escaping for any cookie value usage
   - Use allow-listing for acceptable cookie formats

4. **Security Headers Enhancement**:
   - Remove or obscure rate limiting headers from error responses
   - Consider removing x-powered-by header for technology obfuscation

5. **Monitoring and Logging**:
   - Log suspicious cookie values for security monitoring
   - Implement alerts for repeated malformed cookie attempts
   - Monitor for session fixation attack patterns

6. **Security Testing**:
   - Conduct comprehensive session management security review
   - Implement regular penetration testing for authentication flows
   - Test all cookie handling throughout the application ecosystem

7. **Architecture Review**:
   - Assess how the 'aI' cookie is used throughout the application
   - Ensure proper session management architecture
   - Consider implementing additional anti-fixation measures
