# Information Disclosure via robots.txt Misconfiguration Exposing Internal Infrastructure Details

**ID:** vuln-0008
**Severity:** MEDIUM
**Found:** 2026-03-18 14:01:09 UTC
**Target:** https://einv.cleartax.in
**Endpoint:** /robots.txt
**Method:** GET
**CWE:** CWE-200
**CVSS:** 5.3

## Description

The robots.txt endpoint at https://einv.cleartax.in/robots.txt returns the full application HTML content instead of a proper robots.txt file. This misconfiguration exposes sensitive internal infrastructure details including Sentry Data Source Name (DSN), internal service endpoints, development environment references, and comprehensive Content Security Policy (CSP) configuration with internal service mappings.

The vulnerability was discovered during reconnaissance testing of the ClearTax E-Invoicing platform. When accessing the standard robots.txt endpoint, the server returns the complete application HTML rather than the expected robots.txt file format. This HTML contains detailed security configuration and internal service references that should not be publicly accessible.

## Impact

Exposure of internal infrastructure details provides attackers with valuable reconnaissance information that can be used to:

1. Map internal network architecture and service dependencies
2. Potentially abuse exposed Sentry DSN for error injection or information gathering
3. Identify development and staging environments for targeted attacks
4. Discover internal service naming conventions and authentication patterns
5. Gather intelligence for social engineering or targeted penetration testing

The exposed Sentry DSN (https://b8b593de648a447397559a3e0f33a194@sentry-k8s-prod.internal.cleartax.co/18) could potentially be abused to inject false error reports or gather debugging information from the internal Sentry instance. Internal service endpoints like sentry-k8s-prod.internal.cleartax.co and app-dev-http.clear.in reveal the organization's internal network structure and development practices.

## Technical Analysis

The vulnerability exists due to misconfigured Amazon S3/CloudFront hosting where the robots.txt file either does not exist or is incorrectly configured to serve the default application content.

Technical details exposed in the robots.txt response:

1. Sentry DSN: https://b8b593de648a447397559a3e0f33a194@sentry-k8s-prod.internal.cleartax.co/18
   - Exposes internal Sentry instance endpoint
   - Reveals project identifier and authentication token

2. Internal service endpoints:
   - sentry-k8s-prod.internal.cleartax.co (Sentry monitoring)
   - app-dev-http.clear.in (development environment)

3. Content Security Policy with internal references:
   - connect-src includes app-dev-http.clear.in and sentry-k8s-prod.internal.cleartax.co
   - img-src includes internal S3 buckets (*.s3.ap-south-1.amazonaws.com)
   - Multiple internal service references in CSP directives

4. Development and staging infrastructure:
   - S3 bucket references: einvoice-stage.s3.ap-south-1.amazonaws.com
   - Development environment endpoints

The misconfiguration suggests that either:
- The robots.txt file was not properly configured in the S3 bucket/CloudFront distribution
- Error handling routes all missing files to the main application
- S3 bucket permissions prevent robots.txt access, causing fallback to default content

To reproduce the vulnerability:

1. Access the robots.txt endpoint using curl or web browser:
   curl -s "https://einv.cleartax.in/robots.txt"

2. Observe the response contains HTML content instead of robots.txt format

3. Extract sensitive information from the response:
   - Sentry DSN: https://b8b593de648a447397559a3e0f33a194@sentry-k8s-prod.internal.cleartax.co/18
   - Internal endpoints: sentry-k8s-prod.internal.cleartax.co, app-dev-http.clear.in
   - CSP policy with internal service references
   - S3 bucket paths and development environment details

4. Verify the content type mismatch by checking headers:
   curl -I "https://einv.cleartax.in/robots.txt"
   Expected: Should return Content-Type: text/plain for robots.txt
   Actual: Returns HTML content with incorrect content type

import requests

def test_robots_txt_disclosure():
    """Test for robots.txt information disclosure vulnerability."""
    
    target_url = "https://einv.cleartax.in/robots.txt"
    
    try:
        # Send request to robots.txt
        response = requests.get(target_url, timeout=10)
        
        print(f"URL: {target_url}")
        print(f"Status Code: {response.status_code}")
        print(f"Content-Type: {response.headers.get('Content-Type', 'Not specified')}")
        print(f"Content Length: {len(response.content)} bytes")
        print("\n--- Content Analysis ---")
        
        # Check if response contains HTML instead of robots.txt
        if '<!doctype html>' in response.text.lower():
            print("CRITICAL: robots.txt returns HTML content instead of robots.txt")
            
            # Extract sensitive information
            content = response.text.lower()
            
            # Check for Sentry DSN
            if 'sentry' in content:
                print("\n[+] Sentry DSN/endpoints found:")
                lines = response.text.split('\n')
                for line in lines:
                    if 'sentry' in line.lower():
                        print(f"   {line.strip()[:200]}")
            
            # Check for internal endpoints
            if '.internal.' in response.text:
                print("\n[+] Internal endpoints found:")
                for line in response.text.split('\n'):
                    if '.internal.' in line:
                        print(f"   {line.strip()[:200]}")
            
            # Check for CSP policy
            if 'content-security-policy' in content:
                print("\n[+] CSP policy exposed (may contain internal endpoints)")
                
            return True
            
        else:
            print("INFO: robots.txt returns expected robots.txt content")
            return False
            
    except requests.RequestException as e:
        print(f"ERROR: Request failed - {e}")
        return False

if __name__ == "__main__":

## Remediation

1. Create a proper robots.txt file:
   - Create a standard robots.txt file that adheres to robots.txt specification
   - Place it in the root directory of the S3 bucket/application
   - Ensure it returns Content-Type: text/plain

2. Fix S3/CloudFront configuration:
   - Configure S3 bucket to properly serve robots.txt with correct content type
   - Set up CloudFront error pages to handle missing files appropriately
   - Ensure robots.txt is not caught by application routing rules

3. Remove sensitive information from client-side code:
   - Review and remove internal endpoint references from CSP headers
   - Consider using environment variables for configuration
   - Separate development and production configurations

4. Implement proper error handling:
   - Configure 404 responses to return minimal information
   - Ensure missing static files don't leak application content
   - Implement proper content negotiation

5. Security review:
   - Audit all static file endpoints for similar misconfigurations
   - Review CSP policies for information disclosure
   - Monitor for abuse of exposed endpoints

6. Testing and validation:
   - Test robots.txt endpoint returns proper content
   - Verify Content-Type header is text/plain
   - Ensure no sensitive information is exposed
   - Implement automated checks in CI/CD pipeline
