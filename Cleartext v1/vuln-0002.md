# CSP Bypass Vulnerability - Content Security Policy Not Enforced on accounts.cleartax.in/v2

**ID:** vuln-0002
**Severity:** CRITICAL
**Found:** 2026-03-18 13:24:01 UTC
**Target:** https://accounts.cleartax.in/v2
**Endpoint:** /v2
**Method:** GET
**CWE:** CWE-693
**CVSS:** 9.3

## Description

Discovered that the Content Security Policy (CSP) implementation on ClearTax's authentication system (accounts.cleartax.in/v2) is configured in report-only mode and not enforced, allowing complete bypass of XSS protections. This critical security misconfiguration enables attackers to execute arbitrary JavaScript if any XSS vulnerability is present, completely undermining the intended security controls.

## Impact

Successful exploitation would allow complete bypass of XSS protections, enabling attackers to:
- Execute arbitrary JavaScript in victim browsers
- Steal session cookies and authentication tokens
- Perform account takeover and credential theft
- Exfiltrate sensitive financial and tax data
- Bypass all client-side security controls

The CSP is only implemented as a report-only policy, providing no actual protection against XSS attacks while creating a false sense of security.

## Technical Analysis

The Content Security Policy header is present but configured with 'Content-Security-Policy-Report-Only' instead of 'Content-Security-Policy', which means the policy is not enforced. Testing confirmed that:

1. **Inline Script Execution**: Successfully executed inline scripts without blocking
2. **Eval Function**: eval() operations work without CSP restrictions
3. **No Meta Tag Fallback**: No CSP meta tags exist in HTML source code
4. **Browser Enforcement**: No client-side CSP enforcement detected

The policy exists for monitoring purposes but provides no actual protection. This is particularly dangerous for a financial application handling sensitive tax data, as any XSS vulnerability would be completely unmitigated.

## Remediation

1. **Immediate Enforcement**: Change 'Content-Security-Policy-Report-Only' to 'Content-Security-Policy' in HTTP headers
2. **Policy Review**: Audit and tighten CSP directives to prevent inline scripts and eval usage
3. **Meta Tag Backup**: Implement CSP meta tags in HTML as secondary defense layer
4. **Testing**: Conduct comprehensive XSS testing with enforced CSP
5. **Monitoring**: Continue CSP violation reporting while maintaining enforcement
6. **Browser Compatibility**: Test CSP enforcement across all supported browsers
7. **Documentation**: Update security documentation to reflect enforced CSP policy
