# Information Disclosure via Session Cookie Leakage on Error Responses

**ID:** vuln-0003
**Severity:** MEDIUM
**Found:** 2026-03-18 13:25:52 UTC
**Target:** https://accounts.cleartax.in/v2
**Endpoint:** /* (all endpoints)
**Method:** GET
**CWE:** CWE-200
**CVSS:** 5.3

## Description

The ClearTax SSO system (accounts.cleartax.in/v2) discloses the ssoAnonId session cookie value in HTTP responses for ALL requests, including error responses (404, 405, etc.) to non-existent endpoints. This creates an information disclosure vulnerability where anonymous session identifiers are exposed even when no legitimate authentication activity occurs, enabling user tracking and potential account enumeration.

## Impact

This vulnerability allows attackers to:
- Track unique user sessions across requests and endpoints
- Perform account enumeration by correlating session activity
- Monitor user browsing patterns on the authentication domain
- Potentially correlate anonymous sessions with authenticated users
- Violate user privacy by exposing session identifiers unnecessarily

The ssoAnonId cookie is set with every response, including errors, making session tracking trivial regardless of whether legitimate authentication flows are accessed.

## Technical Analysis

The vulnerability occurs because the SSO system sets the ssoAnonId cookie in the Set-Cookie header for every HTTP response, including:
- 404 responses to non-existent endpoints
- 405 method not allowed responses  
- Error pages and invalid requests

The cookie is properly secured with HttpOnly, Secure, and SameSite=Lax flags, but the issue is that it's being set unnecessarily on error responses. This exposes session identifiers that could be used for:
1. User tracking via unique session IDs
2. Account enumeration if combined with timing or behavioral analysis
3. Session correlation across different user activities

Unlike the main application, the SSO system leaks session identifiers even when no authentication occurs.

## Proof of Concept

To reproduce:

1. Make any request to non-existent endpoint:
   `curl -I https://accounts.cleartax.in/nonexistent-endpoint`

2. Observe response contains:
   `Set-Cookie: ssoAnonId=unique-uuid-here; expires=...; HttpOnly; Path=/; SameSite=Lax; Secure`

3. Repeat with different invalid endpoints:
   `curl -I https://accounts.cleartax.in/random-path-123`
   `curl -I https://accounts.cleartax.in/api/invalid`

4. Note that each response sets a new ssoAnonId cookie even for 404 errors

The cookie is set regardless of endpoint validity or authentication state.

```
import requests
from collections import defaultdict

def test_cookie_leakage():
    base_url = "https://accounts.cleartax.in"
    test_endpoints = [
        "/v2/nonexistent",
        "/invalid-endpoint-123", 
        "/api/does-not-exist",
        "/random/path/here",
        "/v2/404-test"
    ]
    
    session_cookies = defaultdict(list)
    
    for endpoint in test_endpoints:
        url = base_url + endpoint
        try:
            response = requests.get(url, timeout=10, allow_redirects=False)
            
            if 'ssoAnonId' in response.headers.get('Set-Cookie', ''):
                cookie_value = response.headers['Set-Cookie'].split('ssoAnonId=')[1].split(';')[0]
                session_cookies[cookie_value].append({
                    'url': url,
                    'status': response.status_code,
                    'endpoint': endpoint
                })
                print(f"✓ Cookie leaked on {endpoint} (Status: {response.status_code})")
                
        except requests.RequestException as e:
            print(f"Error testing {url}: {e}")
    
    print(f"\nUnique session cookies leaked: {len(session_cookies)}")
    for cookie, requests_list in session_cookies.items():
        print(f"Cookie {cookie[:20]}... appeared in {len(requests_list)} requests")
    
    return session_cookies

# Execute test
test_cookie_leakage()
```

## Remediation

1. **Conditional Cookie Setting**: Only set ssoAnonId cookie on legitimate authentication requests, not error responses
2. **Error Response Filtering**: Remove Set-Cookie headers from 4xx and 5xx error responses
3. **Endpoint Validation**: Validate that cookie setting only occurs on intended authentication endpoints
4. **Monitoring**: Implement logging to detect abnormal cookie setting patterns
5. **Testing**: Conduct comprehensive testing to ensure cookies are only set when appropriate
6. **Privacy Review**: Assess all cookie usage for unnecessary information disclosure
